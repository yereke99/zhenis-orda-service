PROJECT WORKFLOW EXPLANATION
ZHENIS ORDA INSIDE / Genius Orda

Язык документа: русский.
Источник документа: текущий код проекта. Документ не описывает функциональность, которая не найдена в кодовой базе.

1. PROJECT OVERVIEW

1.1. Что это за проект

Проект zhenis-orda-service является Go backend + static frontend MVP для ZHENIS ORDA INSIDE / Genius Orda.

Фактически в коде найдены следующие части:

1. Telegram Bot.
   Реализован в internal/handler/telegram.go.
   Работает через long polling Telegram Bot API.
   Обрабатывает /start, регистрацию Telegram пользователя, выбор языка, диагностику в чате, открытие Mini App через WebApp button, загрузку чеков оплаты и уведомления администраторов.

2. Telegram Mini App.
   Реализован как static SPA:
   - static/index.html
   - static/css/style.css
   - static/js/app.js

   Mini App использует Telegram.WebApp.initData, отправляет его backend в HTTP header X-Telegram-Init-Data и получает user/profile/progress через /api/me.

3. Browser admin panel.
   Реализован в том же static/index.html и static/js/app.js.
   Запускается по URL /admin.
   Использует cookie-based browser session zo_admin_session.
   Backend routes находятся в internal/handler/http.go и internal/handler/admin_http.go.

4. Education/subscription platform.
   В проекте есть 12 уровней, уроки, тесты, задания, тарифы, подписки, платежи, реферальная система, ZHENIS Coin, закрытые каналы, live streams, support messages, broadcast queue и audit log.

5. ZHENIS ORDA / Genius Orda logic.
   Название платформы и брендовые тексты находятся:
   - internal/service/domain.go: BrandTexts()
   - traits/database/database.go: seedSettings()
   - internal/handler/api.go: handlePlatform()
   - static/index.html и static/js/app.js: frontend copy/layout

1.2. Как части работают вместе

1. Пользователь открывает Telegram bot.
2. Bot регистрирует или обновляет пользователя в таблице users через Store.RegisterOrUpdateTelegramUser().
3. Если пользователь пришел по реферальному start payload вида ref..., Store.RegisterOrUpdateTelegramUser() создает связь в users.invited_by_user_id и referrals.
4. Bot показывает главное меню и inline кнопку Mini App.
5. Mini App открывает static/index.html, запускает static/js/app.js, валидирует Telegram initData через backend route /api/me.
6. Пользователь проходит диагностику, выбирает тариф, создает pending payment.
7. Чек оплаты загружается через Telegram bot как document/photo. HTTP upload receipt route в Mini App не найден.
8. Admin открывает /admin, логинится по browser password, видит платежи и вручную approve/reject.
9. При approve backend создает или продлевает subscription, переводит payment в approved, открывает current_level минимум до 1 и начисляет referral rewards.
10. Пользователь получает доступ к LEVEL 1, проходит уроки, отмечает watched, проходит тест.
11. Store.RecalculateUserProgress() и Store.recalculateUserProgressTx() повышают current_level, если все уроки текущего уровня watched и тест уровня passed.

3. APPLICATION STARTUP FLOW

Startup path: cmd/main.go main()

Step 1. Context and signal handling.
main() создает context через signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGINT, syscall.SIGTERM).

Step 2. Config loading.
main() вызывает config.Load().
config.Load() читает environment variables, применяет defaults и вызывает Config.Validate().

Step 3. Logger initialization.
main() вызывает logger.New(cfg.Env).
traits/logger/logger.go выбирает production или development zap config.

Step 4. Database initialization.
main() вызывает database.Open(ctx, cfg.DBPath).
traits/database/database.go:
- создает directory для DB file, если path != :memory:
- открывает SQLite DSN
- включает WAL, foreign_keys и busy_timeout
- ограничивает DB pool до 1 open и 1 idle connection

Step 5. Migrations/table creation.
main() вызывает database.Migrate(ctx, db).
Migrate():
- создает schema_migrations
- выполняет schemaV1
- выполняет indexesV1
- вставляет migration version 1
- вызывает Seed()

Step 6. Seed data.
Seed():
- seedTariffs(): BASIC, STANDARD, VIP
- seedLevels(): 12 levels, lessons, tests, test_questions, test_options, assignment для level 1
- seedSettings(): platform_name, brand_line, stream_name

Step 7. Repository creation.
main() вызывает repository.New(db).

Step 8. Redis/state store initialization.
main() вызывает buildKV(ctx, cfg, log).

buildKV():
- если cfg.RedisAddr пустой: handler.NewMemoryKV()
- иначе создает redis.NewClient()
- Ping(ctx)
- если Redis недоступен: fallback to handler.NewMemoryKV()
- если Redis доступен: handler.NewRedisKV(client)

KV используется для:
- browser sessions
- Telegram diagnostics state
- reminder idempotency keys

Step 9. Handler creation.
main() вызывает handler.NewServer(cfg, store, kv, log).

NewServer():
- stores cfg/store/kv/logger
- создает SessionManager
- создает TelegramInitValidator через service.NewTelegramInitValidator(cfg.Token, cfg.TelegramInitDataMaxAge)

Step 10. Telegram bot creation.
Если cfg.DisableTelegramBot == false:
- handler.NewTelegramBot(cfg.Token, store, kv, cfg.UploadDir, cfg.MiniAppURL, cfg.AdminIDs, cfg.MaxReceiptBytes, log)
- srv.SetBot(bot)
- bot.StartLongPolling(ctx)

Если TOKEN пустой, StartLongPolling() логирует warning и не запускает polling.

Step 11. Scheduler start.
main() вызывает srv.StartSchedulers(ctx).

StartSchedulers() запускает goroutines:
- every 1 minute: runMinuteJobs()
- every 10 minutes: runInactiveReminderJob()
- every 24 hours: runDailyJobs()
- every 1 minute: runLiveStreamReminderJob()

Step 12. HTTP server start.
main() создает http.Server:
- Addr: cfg.Addr()
- Handler: srv.Routes()
- ReadHeaderTimeout 10s
- ReadTimeout 30s
- WriteTimeout 60s
- IdleTimeout 120s

ListenAndServe() запускается в goroutine.

Step 13. Shutdown.
При ctx.Done():
- создает shutdown context 15 seconds
- httpServer.Shutdown()
- logs "shutdown complete"

4. TELEGRAM BOT FLOW

4.1. Bot creation and polling

Файл:
- internal/handler/telegram.go

Functions:
- NewTelegramBot()
- StartLongPolling()
- getUpdates()
- handleUpdate()

StartLongPolling() запускает goroutine, вызывает getUpdates() с timeout 45 секунд и allowed_updates ["message"].

4.2. /start

Handler:
- TelegramBot.handleUpdate()

Flow:
1. Проверяет update.Message != nil и update.Message.From.ID != 0.
2. Читает msg.Text.
3. Если text starts with /start, извлекает startParam из второго слова.
4. Определяет language:
   - ru, если msg.From.LanguageCode начинается с ru
   - иначе kk
5. Вызывает Store.RegisterOrUpdateTelegramUser() с:
   - TelegramID
   - Username
   - FirstName
   - LastName
   - Language
   - StartParam
6. Если сообщение содержит document/photo, переходит в receipt upload flow.
7. Если текст "Қазақша" или "Русский", вызывает Store.SetLanguage() и sendMainMenu().
8. Если есть diagnostics state, обрабатывает ответ диагностики.
9. Если /start или user created:
   - если user.Language пустой: sendLanguageSelection()
   - иначе sendMainMenu()

Important current behavior:
- RegisterOrUpdateTelegramUser() получает language уже как kk или ru, поэтому новый пользователь обычно сразу получает language и language selection может не показаться.
- parseStartPayload() существует, но в handleUpdate() не используется.

4.3. User registration/upsert

Repository:
- internal/repository/users.go
- Store.RegisterOrUpdateTelegramUser()

New user:
- генерирует referral_code = "ref" + telegram_id
- если StartParam начинается с "ref", ищет inviter по users.referral_code
- запрещает self-referral через проверку inviterTelegramID != input.TelegramID
- INSERT INTO users(...)
- если inviter найден, INSERT OR IGNORE INTO referrals(...)

Existing user:
- обновляет username, first_name, last_name, last_seen_at, updated_at
- language меняется только если текущий language пустой
- invited_by_user_id не меняется для существующего пользователя

Tables:
- users
- referrals

4.7. Diagnostics in bot

Functions:
- startDiagnostics()
- handleDiagnosticsText()
- diagnosticsQuestions()
- saveDiagnosticsState()
- getDiagnosticsState()
- diagnosticsKey()

Flow:
1. User sends "Free diagnostics", "Тегін диагностика" или "Бесплатная диагностика".
2. startDiagnostics() сохраняет diagnosticsState в KV по key diag:{userID}.
3. Bot задает 8 вопросов.
4. Каждый ответ сохраняется в state.Answers.
5. После последнего ответа Store.SaveDiagnostics() пишет row в diagnostics.
6. Bot отправляет i18n.T(language, "diagnostics_done") и inline Mini App button.

Tables:
- diagnostics

4.8. Payment receipt upload

Function:
- handleReceiptUpload()

Accepted Telegram message types:
- document
- photo

Allowed extensions:
- .pdf
- .jpg
- .jpeg
- .png
- .webp

Flow:
1. Получает file_id, fileName, mimeType, fileSize.
2. Проверяет fileSize <= maxReceipt.
3. Проверяет extension через allowedReceiptExt().
4. Если extension unknown, пытается mime.ExtensionsByType().
5. downloadTelegramFile() вызывает Telegram getFile, скачивает файл по fileBase/file_path.
6. Сохраняет файл в uploads/receipts/YYYY/MM/{userID}_{unixNano}.{ext}.
7. Store.AttachReceipt() ищет latest pending payment текущего user.
8. INSERT INTO payment_receipts.
9. UPDATE payments SET status='uploaded_receipt', receipt_file_path=...
10. Bot сообщает user payment_uploaded.
11. Bot отправляет admin notifications всем adminIDs.

Tables:
- payments
- payment_receipts

Mini App receipt upload route:
Not found in current codebase.

4.9. Admin notifications

Receipt upload:
- handleReceiptUpload() отправляет "New receipt uploaded..." каждому adminID из config.AdminIDs.

Payment approval/rejection:
- handleAdminApprovePayment() отправляет user i18n payment_approved через s.bot.SendMessage().
- handleAdminRejectPayment() отправляет user formatted rejection message через s.bot.SendMessage().

4.10. User notifications

Found:
- payment_uploaded
- payment_no_pending
- payment_approved
- payment_rejected
- inactive_3_days scheduler reminder
- subscription_ending scheduler reminder
- live stream reminders

Broadcast delivery:
Not found in current codebase. Broadcast records are queued in database, but no scheduler/worker sends them.

5.8. Telegram Mini App mode vs Browser Admin mode vs development mode

- Detected by Telegram.WebApp.initData/initDataUnsafe.user.

- Calls setupTelegramViewport().
- Auth uses X-Telegram-Init-Data.

Browser Admin mode:
- Detected by path /admin.
- Adds browser-admin-mode classes.
- Calls bootBrowserAdmin().
- Auth uses cookie zo_admin_session.
- Does not use Telegram initData.

Development Mini App mode:
- Detected by query ?miniapp_dev=1.
- Adds telegram-miniapp-mode and dev-miniapp-mode classes.
- api() sends X-Miniapp-Dev: 1.
- Optional query params:
  - telegram_id
  - username
- Backend accepts this only when cfg.Env == "development".
- userFromMiniApp() creates/updates dev user with telegramID default 777000.

6.3. forceExpand()

forceExpand():
1. Gets Telegram.WebApp.
2. Calls webApp.ready().
3. Calls webApp.expand().
4. Sets background/header color to #050403 when supported.
5. Calls disableVerticalSwipes() when supported.
6. Calls requestFullscreen() when supported and not already fullscreen.
7. Sets state.fullscreenRequested = true and adds class telegram-fullscreen when requestFullscreen() was requested.
8. Calls syncTelegramFrame() immediately and again after 120ms, 450ms, 900ms.

- --tg-content-safe-area-inset-bottom

6.6. Header below Telegram system buttons

The Mini App header is inside .app-shell.

- safeAreaInset.top
- contentSafeAreaInset.top

This is why user-header should remain below Telegram close/back/system controls instead of being hidden under them.

- Lesson edit/delete buttons in static/js/app.js: Not found in current codebase.

Backend endpoints for lesson create/edit/delete do exist.

9.2. Backend lesson creation flow

Practical backend/API flow:

1. Admin must be authenticated in browser admin.
   Auth route:
   - POST /api/browser-auth/login
   Cookie:
   - zo_admin_session

2. Request must call:
   - POST /api/admin/lessons

3. Expected JSON body maps to repository.Lesson:
   - id: omitted or 0 for create
   - level_id: required
   - title_kk: required by DB schema
   - title_ru: required by DB schema
   - description_kk: optional
   - description_ru: optional
   - video_url: optional
   - sort_order: integer
   - is_active: boolean

Example body:
  {
    "level_id": 1,
    "title_kk": "New lesson title KK",
    "title_ru": "New lesson title RU",
    "description_kk": "Description KK",
    "description_ru": "Description RU",
    "video_url": "https://...",
    "sort_order": 5,
    "is_active": true
  }

4. Backend route:
   - internal/handler/http.go Routes()
   - mux.Handle("POST /api/admin/lessons", admin(s.handleAdminPostLesson, super_admin, content_manager))

5. Middleware:
   - withBrowserAuth()
   - withRole()

6. Handler:
   - internal/handler/admin_http.go handleAdminPostLesson()

7. Handler behavior:
   - decodeJSON(r, &lesson)
   - if path id exists, sets lesson.ID from path
   - if !lesson.IsActive, sets lesson.IsActive = true
   - calls s.store.UpsertLesson()
   - writes audit action "lesson_upsert"
   - returns {"lesson": out}

Important current behavior:
- Because handleAdminPostLesson() does if !lesson.IsActive { lesson.IsActive = true }, clients cannot create/update an inactive lesson through this handler by sending is_active=false. This means "inactive status" exists in DB/repository but HTTP handler forces active on upsert.

8. Repository:
   - internal/repository/admin.go Store.UpsertLesson()

9. If lesson.ID == 0:
   INSERT INTO lessons(level_id, title_kk, title_ru, description_kk, description_ru, video_url, sort_order, is_active)

10. If lesson.ID != 0:
   UPDATE lessons SET level_id=?, title_kk=?, title_ru=?, description_kk=?, description_ru=?, video_url=?, sort_order=?, is_active=?, updated_at=CURRENT_TIMESTAMP WHERE id=?

11. Audit:

Triggered by:
- Store.ApprovePayment()
- applyReferralPaymentRewardsTx(invitedUserID)

Flow:
1. Find referral where invited_user_id = invitedUserID.
2. If none, no action.
3. UPDATE referrals SET status='paid', reward_granted=1 WHERE status='registered'.
4. Add +100 coins to inviter:
   - reason referral_paid
   - source_type referral
   - source_id invitedUserID
5. Count paid/rewarded referrals for inviter.
6. For thresholds, insert referral_rewards.

- 1: 7_days_free
- 3: 1_month_free
- 5: closed_vip_stream
- 10: personal_mini_review
- 20: 1_month_vip_tariff_access
- 50: personal_zoom_with_mentor

Subscription extension:
- threshold 1 grants 7 days free via extendSubscriptionTx()
- threshold 3 grants 30 days free via extendSubscriptionTx()
- thresholds 5/10/20/50 only insert referral_rewards; no further automated action found.

25. FINAL SUMMARY

This project is a Go-based Telegram Bot, Telegram Mini App, and browser admin panel for a subscription education platform called ZHENIS ORDA INSIDE / Genius Orda.

The backend starts in cmd/main.go, loads config, opens SQLite, runs schema/seed migrations, creates repository and HTTP handlers, optionally starts Telegram long polling, and starts background schedulers.

The frontend is a static SPA. static/index.html provides shells for Mini App, admin login, admin panel and landing. static/js/app.js detects Telegram/Admin/Dev modes, sends Telegram initData to backend, renders Mini App screens, and renders the browser admin panel. static/css/style.css defines the dark/gold design system and Telegram safe-area layout.

Users are registered through Telegram bot or Mini App initData. Payments are created in Mini App, receipts are uploaded through Telegram bot, and admins approve/reject payments in /admin. Approved payments activate subscriptions and open LEVEL 1. Users progress through 12 levels by watching all active lessons and passing the active test for the current level. Lessons and tests are stored in SQLite and seeded by traits/database/database.go.

Admin content APIs exist for levels, lessons, tests, channels, streams, settings and broadcasts. However, the current browser admin frontend only has full forms for payment verification, channels and broadcast. Lesson/test creation forms are not found in current frontend code, although backend endpoints exist.
