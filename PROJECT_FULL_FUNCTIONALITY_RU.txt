ПОЛНОЕ ОПИСАНИЕ ФУНКЦИОНАЛА ПРОЕКТА
ZHENIS ORDA INSIDE / Genius Orda

Дата разбора: 2026-05-19
Источник: текущий код проекта zhenis-orda-service.

ВАЖНО:
Этот документ описывает именно то, что найдено в коде. Если какая-то бизнес-идея
не реализована в backend/frontend/SQL, она не записана как готовая функция.


1. ЧТО ЭТО ЗА ПРОЕКТ

zhenis-orda-service - это backend на Go + статический frontend для образовательной
платформы ZHENIS ORDA INSIDE / Genius Orda.

Проект объединяет:

1. Telegram Bot.
   Бот регистрирует пользователя, показывает меню, открывает Telegram Mini App,
   принимает чеки оплаты, ведет диагностику в чате и отправляет уведомления.

2. Telegram Mini App.
   Основной пользовательский интерфейс внутри Telegram. В Mini App пользователь
   видит профиль, прогресс, уровни, уроки, тесты, тарифы, оплату, книги,
   бесплатные уроки, рефералку, монеты, эфиры, каналы и поддержку.

3. Browser Admin Panel.
   Админка открывается по /admin. Через нее админ управляет пользователями,
   платежами, подписками, тарифами, уровнями, уроками, тестами, книгами,
   бесплатными уроками, каналами, эфирами, рассылками, настройками и аудитом.

4. Образовательная платформа.
   В базе есть 12 уровней развития. Пользователь получает доступ после оплаты,
   проходит уроки, отмечает просмотр, сдает тесты, получает ZHENIS Coin и
   постепенно открывает следующие уровни.

5. Платежная и подписочная логика.
   Пользователь выбирает тариф, создает платеж, загружает чек. Админ вручную
   проверяет чек и одобряет или отклоняет платеж. После одобрения создается или
   продлевается подписка.


2. ТЕХНИЧЕСКИЙ СОСТАВ

Основной стек:

- Go backend.
- HTTP server на стандартном net/http.
- Telegram Bot API через long polling.
- Telegram Mini App auth через проверку initData.
- SQLite database с WAL, foreign keys и миграциями.
- Redis для временных состояний, сессий и антидублей. Если Redis недоступен,
  используется in-memory KV.
- zap logger.
- Static frontend без сборщика:
  - static/index.html
  - static/css/style.css
  - static/js/app.js
- Docker / Docker Compose.

Основные папки:

- cmd/main.go - точка входа приложения.
- config/ - загрузка env-конфига.
- internal/handler/ - HTTP routes, Telegram bot, auth, scheduler.
- internal/repository/ - SQL, транзакции, бизнес-правила.
- internal/service/ - чистые helper-функции домена и Telegram auth.
- internal/i18n/ - тексты Telegram bot.
- traits/database/ - SQLite schema, migrations, seed.
- traits/logger/ - настройка логгера.
- static/ - Mini App, landing и admin UI.


3. КАК ЗАПУСКАЕТСЯ СЕРВИС

Логика запуска в cmd/main.go:

1. Создается context с graceful shutdown по SIGINT/SIGTERM.
2. Загружается Config из env и дефолтов.
3. Создается zap logger.
4. Открывается SQLite database.
5. Запускается database.Migrate():
   - создает таблицы;
   - добавляет недостающие колонки;
   - мигрирует старую схему тестов;
   - создает индексы;
   - запускает seed тарифов, уровней и app_settings.
6. Создаются папки:
   - PAYMENT_DIR;
   - UPLOAD_DIR;
   - BOOK_UPLOAD_DIR;
   - FREE_LESSON_UPLOAD_DIR.
7. Создается repository.Store.
8. Создается KV:
   - Redis, если доступен;
   - MemoryKV, если Redis пустой или недоступен.
9. Создается HTTP Server.
10. Если DISABLE_TELEGRAM_BOT=false, создается TelegramBot и запускается long polling.
11. Запускаются background schedulers.
12. HTTP server слушает PORT.
13. При остановке выполняется graceful shutdown.

6. АВТОРИЗАЦИЯ MINI APP

Mini App API защищен middleware withMiniAppAuth.

В production/staging сценарии:

1. Frontend берет Telegram.WebApp.initData.
2. Передает initData в header X-Telegram-Init-Data.
3. Backend проверяет HMAC-подпись Telegram initData через bot token.
4. Проверяется auth_date и max age.
5. Из initData достается Telegram user.
6. Пользователь создается или обновляется в users.

Альтернативно initData можно передать как Authorization: tma <initData>.

Development preview:

Если ENV=development и передан query/header miniapp_dev=1, backend создает
тестового пользователя без Telegram подписи. Можно передать telegram_id,
username, first_name, last_name, photo_url.


7. АВТОРИЗАЦИЯ АДМИНКИ

Админка использует cookie zo_admin_session.

Логика:

1. POST /api/browser-auth/login принимает password.
2. Если ADMIN_PASSWORD_HASH задан, пароль проверяется через bcrypt.
3. Если hash пустой и ENV=development, пароль admin.
4. Создается BrowserSession в Redis/MemoryKV.
5. Cookie HttpOnly, SameSite=Lax, Secure=true только в production.
6. Все /api/admin/* routes требуют эту cookie.

Роли в коде:

- super_admin
- content_manager
- support
- analyst

Важно:
Таблица staff_users есть в базе, но текущий login админки не проверяет staff_users.
После входа через пароль создается session с ролью super_admin.
Role middleware уже написан, но полноценная multi-admin авторизация через
staff_users в текущем коде не реализована.


8. TELEGRAM BOT

Telegram bot находится в internal/handler/telegram.go.

Что умеет бот:

1. Работает через long polling getUpdates.
2. Обрабатывает /start.
3. Создает или обновляет пользователя в users.
4. Принимает start payload для реферальной системы.
5. Показывает выбор языка.
6. Показывает основное меню.
7. Дает inline кнопку открытия Mini App.
8. Запускает диагностику в Telegram-чате.
9. Принимает чек оплаты как document или photo.
10. Скачивает файл из Telegram.
11. Сохраняет чек в PAYMENT_DIR/receipts/YYYY/MM.
12. Привязывает чек к последнему pending/uploaded_receipt платежу пользователя.
13. Уведомляет админов из ADMIN_IDS о новом чеке.
14. Умеет создавать Telegram invite link для каналов/уровней.
15. Есть временная команда /test для админов, если ENABLE_TELEGRAM_TEST_COMMANDS=true.

Меню бота:

- Мой уровень.
- Сабақтарым.
- Тест тапсыру.
- Тапсырмаларым.
- Жабық эфир.
- Дос шақыру.
- Бонустар.
- Төлем мерзімі.
- Қолдау.
- Открытие Mini App.

Замечание:
В коде есть ru i18n map, но тексты в основном казахские. При выборе "Русский"
бот сейчас тоже сохраняет language="kk". Это текущее поведение кода.


9. ПОЛЬЗОВАТЕЛЬСКИЙ СЦЕНАРИЙ: КАК БУДЕТ РАБОТАТЬ

Полный путь пользователя:

1. Пользователь открывает Telegram bot.
2. Нажимает /start.
3. Backend создает user:
   - telegram_id;
   - username;
   - first_name;
   - last_name;
   - language;
   - referral_code вида ref<telegram_id>;
   - current_level=0.
4. Если /start пришел с ref-кодом другого пользователя, создается referral.
5. Бот показывает меню и кнопку Mini App.
6. Пользователь открывает Mini App.
7. Mini App валидирует Telegram initData через backend.
8. Если подписки нет, пользователь видит onboarding/tariffs/payment flow.
9. Пользователь может пройти диагностику.
10. Пользователь выбирает тариф BASIC/STANDARD/VIP.
11. Backend создает pending payment.
12. Пользователь оплачивает через Kaspi QR / Kaspi Pay / Halyk / Bank Card.
13. Пользователь загружает чек:
    - через Mini App;
    - или как файл/фото в Telegram bot.
14. Payment становится uploaded_receipt.
15. Админ видит чек в /admin.
16. Админ вручную одобряет или отклоняет платеж.
17. При одобрении:
    - создается или продлевается active subscription;
    - payment становится approved;
    - current_level пользователя становится минимум 1;
    - если пользователь пришел по рефералке, inviter получает reward;
    - пользователь получает Telegram уведомление.
18. Пользователь открывает уровень 1.
19. Пользователь смотрит уроки и отмечает их как watched.
20. За каждый просмотренный урок начисляется 5 ZHENIS Coin.
21. Пользователь проходит тесты уровня.
22. За успешно сданный тест начисляется 20 ZHENIS Coin.
23. Когда все уроки текущего уровня просмотрены и все активные тесты уровня
    сданы, backend повышает current_level.
24. Пользователь переходит к следующему уровню.
25. Процесс повторяется до уровня 12.

11. ПРАВИЛА ДОСТУПА К УРОВНЯМ

Функции доступа находятся в internal/repository/progress.go.

Пользователь может открыть уровень, если:

1. levelNumber от 1 до 12.
2. У пользователя access_closed=false.
3. У пользователя есть active subscription, у которой expires_at > now.
4. user.current_level >= нужный levelNumber.

После одобрения первого платежа current_level становится минимум 1.

Как открывается следующий уровень:

1. Backend смотрит текущий current_level.
2. Проверяет, что есть active subscription.
3. Проверяет, что в текущем уровне есть активные уроки.
4. Проверяет, что все активные уроки текущего уровня watched.
5. Проверяет, что в уровне есть активные тесты и все они passed.
6. Если все выполнено, current_level увеличивается на 1.
7. Процесс повторяется в цикле, пока пользователь может открыть следующий уровень.

Важная деталь:
Assignments участвуют в расчете процента progress, но не являются обязательным
условием для открытия следующего уровня. Для unlock нужны уроки + тесты.

Если в уровне нет активных уроков или нет активных тестов, следующий уровень не
откроется автоматически.


12. ПРОГРЕСС

Progress содержит:

- level_number;
- total_lessons;
- watched_lessons;
- test_passed;
- assignment_done;
- completed;
- percent;
- next_requirement;
- can_unlock_next;
- subscription_ok.

Расчет:

1. Считаются активные уроки уровня.
2. Считаются просмотренные уроки пользователя.
3. Проверяется, все ли тесты уровня сданы.
4. Проверяется, есть ли отправленное задание.
5. Percent считается как выполненные единицы / все единицы:
   - уроки;
   - тест;
   - задание, если оно есть.

Тексты next_requirement:

- если нет оплаты: "Төлем жасаңыз."
- если подписка закончилась: "Жазылымыңызды ұзартыңыз."
- если не все уроки пройдены: нужно пройти все уроки;
- если тест не сдан: нужно сдать тест;
- если уровень 12 завершен: MASTER завершен, сертификат готовится;
- если все готово: следующий уровень готов к открытию.

5. Пересчитывается progress и возможно открывается следующий уровень.


14. ТЕСТЫ

Тесты привязаны к lesson_id. В базе есть level_id тоже, но актуальная логика
работает через lesson_id и уровень урока.

Admin может создать один тест на урок, потому что есть unique index по lesson_id.

Test содержит:

- id;
- level_id;
- lesson_id;
- title;
- pass_percent;
- is_active;
- questions;
- options.

Правила теста:

1. pass_percent должен быть от 1 до 100.
2. В тесте должен быть минимум один вопрос.
3. В каждом вопросе минимум два варианта.
4. В каждом вопросе ровно один correct option.
5. Пользователю correct flags не отдаются.
6. При submit backend считает correct_count и score_percent.
7. passed=true, если score_percent >= pass_percent.
8. Успешная сдача дает 20 ZHENIS Coin.
9. Для открытия следующего уровня должны быть сданы все активные тесты уровня.

Пользовательский API:

- GET /api/tests/{level_id}
- POST /api/tests/{level_id}/submit

17. ПЛАТЕЖИ И ЧЕКИ

Payment statuses:

- pending;
- uploaded_receipt;
- approved;
- rejected;
- expired;
- cancelled.

Сценарий payment:

1. Пользователь выбирает тариф.
2. POST /api/payments создает pending payment.
3. amount_kzt берется из тарифа.
4. expires_at = now + PAYMENT_PENDING_TTL_MINUTES.
5. Пользователь получает payment instructions.
6. Пользователь загружает чек.
7. Backend сохраняет файл.
8. Backend парсит чек насколько возможно.
9. Создается payment_receipts.
10. Payment переводится в uploaded_receipt.
11. Admin принимает решение.

Загрузка чека возможна:

- через Mini App: POST /api/payments/{id}/receipt, multipart field receipt;
- через Telegram bot: document или photo.

Допустимые форматы чеков:

- PDF;
- JPG/JPEG;
- PNG;
- WEBP.

Receipt validation:

- считается SHA256 файла;
- из PDF пытается вытащить printable text;
- image OCR сейчас не реализован;
- QR decode сейчас не реализован;
- ищется сумма;
- ищется валюта;
- ищутся transaction/check/reference IDs;
- ищется дата платежа;
- определяется provider по тексту;
- проверяется amount mismatch;
- проверяется дата раньше payment.created_at - 24 часа;
- проверяются дубликаты по file_hash, raw_text_hash, qr_payload_hash,
  parsed_transaction_id, parsed_check_id.

Receipt validation statuses:

- uploaded;
- parse_failed;
- parse_partial;
- valid_candidate;
- suspicious;
- duplicate;
- rejected;
- approved.

Важное ограничение:
OCR для изображений и QR decoding сейчас являются stub-функциями. Поэтому фото
чека чаще будет уходить в ручную проверку.


18. ОДОБРЕНИЕ И ОТКЛОНЕНИЕ ПЛАТЕЖА

Admin approve:

1. Проверяет, что payment pending или uploaded_receipt.
2. Если чек duplicate или suspicious, approve требует super_admin и comment.
3. Если чек parse_failed или parse_partial, approve требует comment.
4. Если у пользователя уже есть active subscription, она продлевается от текущей
   даты окончания.
5. Если active subscription нет, создается новая.
6. subscription_days берется из запроса или SUBSCRIPTION_DEFAULT_DAYS.
7. payment становится approved.
8. latest receipt получает validation_status=approved.
9. user.current_level становится минимум 1.
10. Применяются referral rewards.
11. Пользователь получает Telegram уведомление об одобрении.

Admin reject:

1. Нельзя отклонить уже approved payment.
2. payment становится rejected.
3. latest receipt получает validation_status=rejected.
4. admin_comment сохраняется в payment.
5. Пользователь получает Telegram уведомление с причиной.

Background job каждую минуту:

- pending платежи с истекшим expires_at переводит в expired;
- active subscriptions с истекшим expires_at переводит в expired.

20. РЕФЕРАЛЬНАЯ СИСТЕМА

Каждый user получает referral_code:

- ref<telegram_id>

Referral link:

- https://t.me/zhenisOrdaFinanceBot?start=<referral_code>

Когда новый пользователь приходит по ref-коду:

1. Backend ищет inviter по users.referral_code.
2. Self-referral запрещен.
3. invited_by_user_id записывается в users.
4. Создается referrals со status=registered.

Когда приглашенный пользователь оплачивает и payment approve:

1. Referral переводится registered -> paid.
2. Inviter получает 100 ZHENIS Coin.
3. Считается paid referral count.
4. Выдаются threshold rewards.

Threshold rewards:

- 1 оплаченный друг: 7 дней бесплатно.
- 3 оплаченных друга: 1 месяц бесплатно.
- 5 оплаченных друзей: закрытый VIP эфир.
- 10 оплаченных друзей: личный mini-review.
- 20 оплаченных друзей: 1 месяц VIP tariff access.
- 50 оплаченных друзей: личный Zoom с ментором.

Реально автоматизировано:

- threshold 1 продлевает/создает BASIC subscription на 7 дней;
- threshold 3 продлевает/создает BASIC subscription на 30 дней;
- остальные rewards записываются как granted, но без отдельной автоматической
  выдачи доступа.

29. НАПОМИНАНИЯ И ФОНОВЫЕ ЗАДАЧИ

Schedulers запускаются в Server.StartSchedulers().

Задачи:

1. Каждую минуту:
   - expire pending payments;
   - expire subscriptions;
   - expire level telegram invites.

2. Каждые 5 секунд:
   - обработка queued broadcasts.

3. Каждые 10 минут:
   - поиск пользователей, не заходивших 72 часа;
   - отправка inactive reminder;
   - антидубль через KV key reminder:inactive3:<userID>.

4. Каждые 24 часа:
   - поиск подписок, которые заканчиваются примерно через SUBSCRIPTION_REMINDER_HOURS;
   - отправка subscription ending reminder.

5. Каждую минуту:
   - напоминания по live streams за 24h/3h/30m.

35. КОРОТКО: КАК ЭТО ДОЛЖНО ВЫГЛЯДЕТЬ ДЛЯ БИЗНЕСА

Это не просто "курс". Это Telegram-based платформа закрытого клуба:

1. Пользователь заходит через Telegram.
2. Видит брендированный Mini App.
3. Проходит диагностику и Financial IQ.
4. Покупает тариф.
5. Загружает чек.
6. Админ подтверждает оплату.
7. Пользователь получает доступ к уровню 1.
8. Каждый уровень состоит из уроков и тестов.
9. После выполнения условий открывается следующий уровень.
10. Пользователь набирает ZHENIS Coin.
11. Пользователь зовет друзей и получает бонусы.
12. По тарифу и уровню открываются закрытые каналы и эфиры.
13. Админ управляет контентом, оплатами, пользователями и рассылками из /admin.

Итоговая логика курса:

- Оплата открывает вход.
- Подписка удерживает доступ.
- Уроки + тесты двигают пользователя по 12 уровням.
- Coins, referrals, закрытые каналы и эфиры добавляют мотивацию и клубность.
- Admin panel закрывает операционку: контент, платежи, пользователи, рассылки,
  настройки и аудит.
